# Email Watch

## Decision

An email watcher is an `EmailWatcherRecord` (per `(account, sender-query)`) on
`RuntimeState`. Each watcher is driven by a backing interval-driven scheduled job
whose **pre-run hook** is the generic `skill-script` handler (see
[Pre-LLM Job Hooks](job-pre-run-hooks.md)) routed at a bundled, in-tree detection
script (`skills/google/gmail-watch/scripts/detect.ts`). The script runs HEADLESS
(no agent turn, no approval): it polls the user's Gmail through the existing `gws`
CLI for new matching message ids, dedups them against a small caller-supplied
state, applies a deterministic safety floor, and returns the RAW metadata of each
surviving new match as an untrusted item — the trusted hook RUNNER fences it. The
hook maps the result onto the scheduler's typed contract — **zero matches →
short-circuit (no model turn); one or more → the drafting turn runs in the
watcher's dedicated chat session with the fenced matches injected as context**.
The woken agent reads the full message and composes/sends a reply via the
EXISTING `google-gmail` skill — the detection script reads ONLY metadata
(From/Subject/Date/snippet) and never message bodies.

The architecture is deliberately a **thin deterministic detection floor + skill-driven action**, riding the general pre-run job-hook primitive with the detection floor itself a provisioned, trusted SKILL SCRIPT (not bespoke runtime code):

- **Trigger floor (the `gmail-watch/detect` skill script via the `skill-script` hook):** poll `gws` → dedup → safety-filter → return raw matches. Zero model turns when there's no new matching mail (the run short-circuits before the model).
- **Action (reuse the skill):** the drafting turn reads the body and drafts/sends via `google-gmail`, gated by `terminal_exec`'s existing approval. The action playbook lives in the backing job's prompt (outside the runner's untrusted fence).

No native Gmail REST client, no new OAuth flow, no new token store, and no new
`gmail_send`/`gmail_draft` tools are introduced. The watcher only writes a
config record; everything side-effecting flows through the already-approval-gated
skill.

## Context

This follows the messaging-bridge ADRs ([telegram-bridge.md](telegram-bridge.md),
[discord-bridge.md](discord-bridge.md)) and the connector-reprobe maintenance
loop. The user asked for an email-watching feature with multi-account support,
sender filtering, and drafted responses, and chose to reuse the already-set-up
`gws` (`google-gmail`) skill rather than build a native Gmail client.

`gws` holds ONE signed-in Google identity per machine, so v1 watches that single
account. The data model is multi-account-SHAPED (`provider` + `accountEmail` +
`credentialName` per record) so a later phase can fan out to per-account
credentials (gws multi-profile or native tokens) without a schema migration.

Reference systems shaped the design: OpenClaw's untrusted-content wrapping (borrowed),
Hermes's layered inbound-safety + startup watermark seeding (borrowed), and the
"separate email-as-connector from watching-as-scheduler" lesson from Claude Code.
OpenClaw's Pub/Sub-push infrastructure and Hermes's auto-send + plaintext creds
were rejected as the wrong shape for a local-first, consent-gated runtime.

## Required Now

- `EmailWatcherRecord` carries `{ id, instance, agentId?, provider:"gmail", accountEmail?, credentialName?, query, labelIds?, chatSessionId?, jobId?, enabled, status:"ok"|"error"|"needs_auth", lastError?, lastPolledAt?, createdAt, updatedAt }`. `jobId` back-references the backing scheduled job that drives the watcher; the detection cursor/dedup are NOT on this record (they live on the job). State CRUD lives in `src/state/email-watchers.ts`; all writes go through `mutateState`, never the JSON file directly.
- Detection state (the cursor + a tiny boundary dedup set) lives on the backing `JobRecord.hookState` (job-owned; see [Pre-LLM Job Hooks](job-pre-run-hooks.md)), NOT in a separate table. The script is a PURE function `{query, state} -> {result, newState}`; the scheduler persists `newState` at the at-least-once commit boundary. There is no `email_seen` SQLite table — the `after:<sec>` watermark keeps the `seen` set bounded to one second's worth of boundary ids, so a small in-state set replaces the per-message table. Removing a watcher removes its backing job, which drops the detection state with it.
- Each watcher's backing job is an interval-driven scheduled job (60s) bound to the watcher's dedicated chat session, with a `preRunHook = { handlerId: "skill-script", config: { skill: "gmail-watch", script: "detect", watcherId, query, account? } }`. The single scheduler (`runDueJobs`, the 1s self-rescheduling loop) is the only driver — there is no dedicated gmail loop and no `GINI_GMAIL_POLL_MS` knob; the cadence is the job's `intervalSeconds`. The in-flight hook + its drafting turn run inside `runDueJobs`, already covered by the existing SIGTERM scheduler drain. Creating a watcher provisions the job (rolled back if job creation fails); removing it removes the job AND the dedicated chat session; disable/enable pause/resume it. A startup `backfillEmailWatcherJobs` provisions a backing job for any enabled watcher missing a resolvable `jobId` (idempotent self-heal). Provisioning is idempotent against the crash-between-create-and-stamp window: before creating, both `addEmailWatcher` and `backfillEmailWatcherJobs` ADOPT an existing job whose `preRunHook.config.skill === "gmail-watch"` + `watcherId` matches (re-stamping `jobId`) instead of creating a duplicate, so a watcher never ends up double-polling.
- Before polling, the `detect` script checks `gws auth status`. When signed out, it returns a `shortCircuit` carrying `status: "needs_auth"` in its opaque state, with the prior `cursor`/`seen` UNCHANGED (no spam, NO model turn); the backing job stays active and the next tick retries once the user re-auths. Signed-out is NEVER a non-zero script exit / hook `error`, which would fail the job and stop scheduling.
- The script spawns `gws` exactly as `gws-session.ts` does: `zsh -lc`, `stdin: "ignore"`, kill-on-timeout, inheriting the env the skill-script runner provides (`PATH`/`HOME` for the login shell + `GINI_*`). `gws` reads its own client config + token from `~/.config/gws/`, so no credentials are injected. stdout AND stderr are drained CONCURRENTLY (an unread piped stream can fill its ~64KB OS buffer and deadlock the child until the kill timer fires). The gws subprocess boundary is injectable so the unit test stubs it. `gws` prints a `Using keyring backend:` preamble to STDERR before its JSON, so stdout begins at the first `{`; the parser still defensively skips to the first `{`.
- Detection uses `gws gmail users messages list --params '{"userId":"me","q":"<query>","maxResults":N}' --format json --page-all --page-limit P`. Gmail lists newest-first, so for a steady-state window that fits within the page cap the script enumerates the WHOLE matching window across pages (NDJSON, one object per page) and drains it oldest-first rather than the newest page only — otherwise older matches that don't fit on the first page would fall below the advancing watermark. When the page cap IS hit the window isn't fully enumerated, and the seeding/truncated-window handling below baselines past the un-listed tail rather than draining it. Once a watcher has a watermark the query is bounded with `after:<epochSec>` derived from the state cursor, so steady-state polling lists almost nothing.
- Metadata for the safety floor + match item comes from `gws gmail users messages get --params '{"userId":"me","id":"<id>","format":"metadata","metadataHeaders":["From","Subject","Date"]}'`. Bodies are NEVER fetched here.
- Safety floor (deterministic, drop-at-trigger): drop automated senders (From matches `no-reply` / `noreply` / `mailer-daemon` / `postmaster` / `bounce` / `notifications@`) and drop self (the address parsed out of the From header equals the signed-in address from `gws gmail users getProfile`, compared by EQUALITY — not substring, which would false-drop a human whose address merely contains self's). Dropping is deterministic, so re-evaluating a dropped message on a retry is harmless and needs no separate dedup.
- On a surviving match the script emits the RAW metadata (`{from,subject,date,id,snippet}` as a single JSON string) as an `untrusted:true` item — it does NOT fence it. The hook returns the items as `context`; the RUNNER fences each one (below). The scheduler then runs ONE drafting turn for the watcher's job (`dispatchPromptRun`, `mode: "chat"` bound to the watcher's session) with all surviving matches for that tick injected as context — bounded by `MAX_MESSAGES_PER_TICK`, so one tick injects at most that many fenced matches into one turn. The trusted action playbook (`read_skill google-gmail`, read the full message, post a PROPOSED reply in this chat, do NOT send unless asked, emit exactly `[SILENT]` when nothing is actionable) lives in the backing job's prompt, OUTSIDE the untrusted fence. When the script returns zero matches the run short-circuits with no model turn at all.
- The matched metadata is fenced by the RUNNER (reviewed core code), not the script — the prompt-injection boundary is one place, not per-handler (see [Pre-LLM Job Hooks](job-pre-run-hooks.md)). The PRIMARY breakout defense is the JSON encoding: the runner emits each untrusted item's text as a single JSON string, so quotes/newlines/markers are escaped and the whole payload stays on one physical line and cannot break the container even if a sentinel-like substring survives. Belt-and-suspenders: the payload is stripped of the fence sentinel (the strip loops to a fixpoint, so a nested rejoin cannot re-form a sentinel) and has CR/LF collapsed, the close delimiter carries a per-item nonce (derived deterministically from the text) so the close token can't be guessed and forged, and the payload is truncated to the char cap BEFORE the close marker is appended so an oversized payload can't push the marker out.
- Startup seeding BASELINES, it does not enumerate. The `after:` watermark only ever moves FORWARD (toward newer mail), so an older un-listed tail can never be reached by advancing it — a seeding path that depends on enumerating the whole backlog would be wrong on a large inbox (gws lists only the newest page-cap worth). Instead, on a watcher with no cursor the first fire takes the NEWEST listed id, fetches only its `internalDate`, sets the cursor there, records just that one boundary id (plus any sibling sharing its exact epoch second, since `after:` is inclusive of the boundary second) in `seen`, and emits NO item — so the run short-circuits with no model turn (or, if the inbox had no matches, baselines the cursor at `now`). Pre-existing mail older than the baseline is excluded by `after:` forever — which is the correct behavior (never draft a backlog), regardless of inbox size, with no pagination dependence on first run.
- Steady-state draining of a fully-enumerated (non-truncated) window is OLDEST-FIRST. Each fire enumerates the window, skips ids in the boundary `seen` set, drains the rest oldest-first, caps the matches at `MAX_MESSAGES_PER_TICK = 25`, advances the cursor ONCE to the LAST CONSUMED item's `internalDate`, and recomputes `seen` to the ids sharing that new cursor's exact second — forward progress that drains a >cap backlog over successive ticks without ever stepping past an un-consumed match.
- A TRUNCATED steady-state window (more genuinely-new matches than the page cap of `WINDOW_PAGE_LIMIT * WINDOW_PAGE_SIZE` since the cursor — only reachable after downtime + high volume) is a **bounded baseline-with-notice, not an eventual drain**. Because the older tail isn't listed and the forward-only watermark can never reach it, the script does NOT silently advance past the tail and does NOT storm a draft per message: it jumps the cursor to the NEWEST listed match (falling back to `now` if its metadata-get is bad, so the cursor always advances) and returns a `shortCircuit` whose NON-silent `summary` tells the user a large backlog accumulated and that it isn't drafting replies to all of them (it can triage on request). Because a shortCircuit's state persists IMMEDIATELY, the advanced cursor excludes the backlog next tick — at most one notice per episode. Zero overflow drafts, no re-loop.
- Delivery is **exactly once on the happy path, at-least-once across the delivery boundary**, owned by the job-state commit TIMING (not a per-match thunk): the `detect` script returns the advanced cursor + boundary `seen` on a `context` result, and the scheduler persists that `newState` ONLY after `dispatchPromptRun` has spawned the drafting turn. A throw before delivery (a mid-poll list/get failure leaves the OLD state via the script's error-shortCircuit-with-unchanged-state, or a `dispatchPromptRun` spawn failure after the script returned, which skips the deferred state commit) leaves the cursor where it was, so the matches re-detect on a later healthy fire. ShortCircuit results (seeding baseline, no-match, dropped-only, the truncated backlog notice) persist `newState` immediately, since there is nothing to deliver.
- A gws/transport failure inside the script is reported as a `shortCircuit` carrying `status: "error"` + a SCRUBBED `lastError` in its opaque state, with the `cursor`/`seen` UNCHANGED — never a non-zero exit, because the `skill-script` handler turns a non-zero exit into a throw that the runner classes TRANSIENT and a config error would flip `job.status="failed"`. The backing job keeps scheduling and the next healthy tick re-detects from the unchanged cursor (a clean tick emits `status: "ok"` and clears `lastError`). The scrub redacts credential-bearing `.json`/`.enc` files and any home-rooted (`/Users/…`, `/home/…`, `/root`) path so a leaked diagnostic can't expose the filesystem layout.
- **Watcher health is surfaced via job-owned state, derived on read.** The generic `skill-script` handler can't write the watcher record, so the detection script records the last tick's health (`status` + scrubbed `lastError`) in its opaque state blob, which the scheduler persists onto the backing `JobRecord.hookState` each tick (a `shortCircuit` persists immediately, so the signed-out/error/healthy status lands every tick). The email read path then DERIVES the displayed status: `listEmailWatchers` / `getEmailWatcher` (the canonical path the `email_watch` tool, `GET /api/email/watchers`, and `gini email list` use) look up the backing job by `watcher.jobId` and overlay `hookState.status` / `hookState.lastError` onto the returned record. `EmailWatcherRecord.status` / `lastError` are thus derived-on-read; `enabled` stays the separate lifecycle flag, and a watcher with no backing-job hookState (legacy / pre-first-tick) keeps its stored status. This is email-domain code only — `src/hooks` and the generic `skill-script` handler are untouched.
- The `email_watch` agent tool (toolset `email`, action enum `add | list | remove | disable | enable`) is always-on and low-risk — it only writes a watcher record (and, on `add`, a dedicated chat session). `disable` / `enable` pause/resume a watcher (and its backing job) without deleting it. One tool with an action enum keeps the catalog surface minimal, mirroring the `create_job` / `list_jobs` / `delete_job` precedent collapsed into one.
- `add` builds the query: a raw `query` wins; else `from:<sender> is:unread`; else `is:unread`. The watcher + its dedicated chat session are created in ONE `mutateState` write (no orphan session on failure), then `addEmailWatcher` provisions the backing job and stamps `EmailWatcherRecord.jobId`; if job creation fails the watcher is rolled back so no orphan watcher/session lingers. `remove` deletes the backing job first (so the scheduler stops firing it AND drops the job-held detection state), then the watcher row + its dedicated chat session (an auto-created job channel that must not outlive the watcher — `deleteChatSession` cascades its messages/blocks/identity snapshot, guarded if the session is already gone). The tool, `POST /api/email/watchers`, and `gini email add` all route through the shared `addEmailWatcher` helper so they produce identical records.
- `disable` / `enable` route through `setEmailWatcherEnabled`, which flips `EmailWatcherRecord.enabled` AND pauses/resumes the backing job (`updateJobStatus`) so the scheduler stops/starts claiming it. A paused job is never claimed, so the detection script never runs while disabled.
- HTTP: `GET /api/email/watchers` (list), `POST /api/email/watchers` (add), `PATCH /api/email/watchers/:id` (toggle `{ enabled }`), `DELETE /api/email/watchers/:id` (remove) — thin handlers over the state module. CLI: `gini email list | add --from <sender> [--query <q>] | remove <id> | disable <id> | enable <id>`.

## Trust Boundary

- Inbound email is untrusted external content. The detection script emits RAW matched metadata as `untrusted:true` items; the trusted hook RUNNER (reviewed core code) fences each one with the JSON+nonce+fixpoint hardening before it reaches the model, so a hostile subject/snippet cannot smuggle instructions and the script never has to get the fence right (see [Pre-LLM Job Hooks](job-pre-run-hooks.md)). Bodies are read only inside the drafting turn, through the consent-gated `google-gmail` skill — the deterministic script never touches them. The script runs HEADLESS, outside the spawned task's approval/audit/trace envelope, and is PURE — it writes no state itself (the cursor rides back on its result for the scheduler to persist).
- Drafting a reply is surfaced as a CHAT MESSAGE in the watcher's session for the user to review; the agent does not auto-send. Sending requires an explicit user instruction, which then runs `gws gmail +reply` through `terminal_exec`'s approval gate — no new send tool or risk plumbing is added.
- The script shells `gws` with fixed query strings and integer `after:` values it builds itself; raw email bytes never reach the shell. `gws` reads its own client config + token from `~/.config/gws/`, so no credentials are injected into the spawn env and none land on the watcher record. The hook config carries only declarative data (the query + the skill/script routing keys), never executable code — the detection logic is the in-tree, reviewed `gmail-watch/detect` script.

## Limitations / Open Questions

- **Single account (v1).** `gws` holds one signed-in identity, so v1 watches that account. The data model is multi-account-shaped; true multi-account needs gws multi-profile or native per-account tokens (a future phase).
- **`after:`-second cursor precision.** Gmail's `after:` is second-granular while the watermark is millisecond `internalDate`. The boundary `seen` set (the ids sharing the cursor's exact second, carried in the job state) makes the second-rounding harmless: a re-listed boundary message is skipped. The set is bounded to one second's worth of mail by the watermark.
- **History robustness.** v1 uses `messages.list` + a per-watcher watermark rather than `history.list`, so there's no `historyId`-404 full-resync path to maintain. If a future phase moves to `history.list` for lower latency/cost, it must add the bounded full-resync on a 404 history id.
- **Drafts in Gmail.** v1 posts the proposed reply as a chat message, not a Gmail draft. Landing drafts in the user's real Gmail Drafts (threaded) is a future option.
- **Pub/Sub-pull low-latency.** Polling at ~60s is the v1 cadence. A future opt-in could use Pub/Sub-pull (outbound only, no public webhook) for lower latency without changing the trust model.

## Verification

- `bun test skills/google/gmail-watch/scripts/detect.test.ts` exercises the stateless detection engine with an injected `gwsSpawn` (state in, state out, no store): the parse helpers (the `gws` preamble + multi-page NDJSON window, auth-status, metadata), the error scrub (`.json`/`.enc` files, home-rooted + `/root` paths, no over-redaction of `/rootcause`), the safety floor (automated + self by address equality, no false-drop of a human whose address contains self's), the raw (un-fenced) match item, the watcher-state regimes — seeding baseline + same-second siblings, dedup at the boundary second, the oldest-first backlog drain + cap (cursor advances to the last consumed item, the remainder drains on later fires), the truncated-window backlog notice (a non-silent shortCircuit, cursor jumps to newest, advances even on a bad newest metadata-get, no re-loop), the `after:<epochSec>` watermark bound (present once a cursor exists, absent on seeding), a clean tick stamping `status: "ok"`, and at-least-once (re-running with the old state re-detects) — plus the `run()` wrapper's health mapping (signed-out → `needs_auth`, gws-error → `error` + scrubbed `lastError`, both with cursor/seen unchanged; a healthy tick → `ok`).
- `bun test src/capabilities/skill-script-hook.test.ts` covers the generic `skill-script` handler that runs the detect script: the stdout → `HookResult` mapping, the state round-trip, and the error taxonomy (missing skill/script → config error; non-zero exit → transient throw).
- `bun test src/hooks` covers the general pre-run hook primitive and its module boundary, including the runner's untrusted fence + the job-owned `hookState` commit timing (see [Pre-LLM Job Hooks](job-pre-run-hooks.md)). `bun test src/state` covers the `EmailWatcherRecord` CRUD + the backing-job lifecycle (provision/adopt/remove/disable-enable) and the derive-on-read health (a `needs_auth`/`error`/`ok` hookState surfaces on `listEmailWatchers`/`getEmailWatcher`, with the scrubbed `lastError`; a healthy tick clears a prior error; a watcher with no hookState keeps its stored status).
